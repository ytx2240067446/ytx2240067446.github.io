ret2syscall

  • 控制程序执行系统调用来获取 shell
  • Linux 32 位系统调用通过 int 80h 实现，用系统调用号来区分入口函数
    • 64 位系统调用通过 syscall 来实现
  • 操作系统实现系统调用的基本过程是：
    1. 应用程序调用库函数（API）；
    2. API 将系统调用号存入 EAX，然后通过中断调用使系统进入内核态；
    3. 内核中的中断处理函数根据系统调用号，调用对应的内核函数（系统调用）；
    4. 系统调用完成相应功能，将返回值存入 EAX，返回到中断处理函数；
    5. 中断处理函数返回到 API 中；
    6. API 将 EAX 返回给应用程序
  • Linux 系统调用号定义在 ./include/asm/unistd_32.h 和 ./include/asm/unistd_64.h 中

例子

  • 惯例检查安全性，开启了栈不可执行

  • 检查程序，发现 s 和上面两题的偏移量还是一样的
  • 为了触发系统调用，我们需要吧 sh 的路径存入通用寄存器里，然后利用 syscall
  • 比如利用如下系统调用方式来获取 shell
  • 32 位程序的系统调用方式为：
    • eax 存储系统调用号，这里需要 execve 函数，则为 11
    • ebx 存储第一个参数，这里是 “/bin/sh”
    • ecx 存储第二个参数，这里是 NULL，也就是0
    • edx 存储第三个参数，内容同上

  • 64 位程序的系统调用对应的换成了 rax, rdi, rsi, rdx

  • 我们需要借助程序中的汇编代码对寄存器的值进行构造，使其刚好储存可以进行系统调用时的值

    • 比如找到一个栈顶为 11 ，且汇编代码为 pop eax 的情况，此时执行完 pop eax 后 eax 就会为 11
  • 这里可以利用工具 ROPgadget
  • 首先寻找能控制 eax 的汇编片段
  • 上述几个函数都可以控制 eax，随便选一个进行构造就行，注意构造的内容，选第二个的话栈不会被影响
  • 同样寻找 ebx 的片段
  • 发现 0x0806eb90 的非常契合我们的需要，因为能控制到 ebx, ecx, edx 的内容
  • 然后寻找 “/bin/sh” 字符串的地址
  • 还需要一个 int 0x80 的地址来触发系统中断

    • 也就是 pop eax ; pop ebx ; pop esi ; pop edi ; ret

ret2libc3

  • 文件 ret2libc3

  • 程序还是差不多，s 的偏移量还是一样的，只是这次没有 system，也没有 /bin/sh

  • 其实 system 函数属于 libc.so 这个动态链接库中的
    • 而这个库中函数之间相对偏移是固定的
    • 即使 Linux 开启了 ASLR 保护，地址最低的 12 位还是不会变化的
  • 如果知道了 libc 中某个函数的地址，可以借助这个地址在 libc 数据库中寻找对应版本的 libc.so
    • 然后我们就可以获取 system 函数的地址了

  • 如果需要知道 libc 中某个函数的地址，可以采用 got 表泄漏的方式

    • 由于 libc 的延迟绑定机制，我们需要泄漏已经执行过的函数的地址

  • 借助 LibcSearcher 可以快速的查找到对应版本 libc 的信息

  • 此外，在得到 libc 之后，其实 libc 中也是有 /bin/sh 字符串的，所以我们可以一起获得 /bin/sh 字符串的地址

  • 那么只需要泄漏 libc 中某个函数的地址，查出来对应 system 的偏移地址，还有 /bin/sh 的地址
    • 再次执行程序后栈溢出执行 system(“/bin/sh”) 就行了

  • 我们可以利用 puts 函数打印出 __libc_start_main 的地址，然后返回 _start 再执行程序，获取 shell
