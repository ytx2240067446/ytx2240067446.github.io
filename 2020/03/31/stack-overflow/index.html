ret2syscall

  • 控制程序执行系统调用来获取 shell
  • Linux 32 位系统调用通过 int 80h 实现，用系统调用号来区分入口函数
    • 64 位系统调用通过 syscall 来实现
  • 操作系统实现系统调用的基本过程是：
    1. 应用程序调用库函数（API）；
    2. API 将系统调用号存入 EAX，然后通过中断调用使系统进入内核态；
    3. 内核中的中断处理函数根据系统调用号，调用对应的内核函数（系统调用）；
    4. 系统调用完成相应功能，将返回值存入 EAX，返回到中断处理函数；
    5. 中断处理函数返回到 API 中；
    6. API 将 EAX 返回给应用程序
  • Linux 系统调用号定义在 ./include/asm/unistd_32.h 和 ./include/asm/unistd_64.h 中

例子

  • 文件 ret2syscall

  • 惯例检查安全性，开启了栈不可执行

  • 检查程序，发现 s 和上面两题的偏移量还是一样的
  • 为了触发系统调用，我们需要吧 sh 的路径存入通用寄存器里，然后利用 syscall
  • 比如利用如下系统调用方式来获取 shell
  • 32 位程序的系统调用方式为：
    • eax 存储系统调用号，这里需要 execve 函数，则为 11
    • ebx 存储第一个参数，这里是 “/bin/sh”
    • ecx 存储第二个参数，这里是 NULL，也就是0
    • edx 存储第三个参数，内容同上

  • 64 位程序的系统调用对应的换成了 rax, rdi, rsi, rdx

  • 我们需要借助程序中的汇编代码对寄存器的值进行构造，使其刚好储存可以进行系统调用时的值

    • 比如找到一个栈顶为 11 ，且汇编代码为 pop eax 的情况，此时执行完 pop eax 后 eax 就会为 11
  • 这里可以利用工具 ROPgadget
  • 首先寻找能控制 eax 的汇编片段

上述几个函数都可以控制 eax，选一个合适的进行构造，注意构造的内容，选第二个的话栈不会被影响

  • 注意，不能选 0x0804f704 的片段，因为 x86 函数规定了：
    • 返回值不超过 4 字节(如 int、short、char、指针 等类型)，将其保存在 eax 寄存器中，调用方通过读取EAX获取返回值
    • 返回值大于4字节而小于8字节(如 long long 或 __int64 类型)，则通过 eax+edx 寄存器联合返回，其中 edx 保存返回值高4字节，eax 保存返回值低4字节
    • 若返回值为浮点类型(如 float 和 double)，则通过专用的协处理器浮点数寄存器栈的栈顶返回
    • 若返回值为 struct 或 union，则主调函数向被调函数传递一个额外参数，该参数指向将要保存返回值的地址。即函数调用 func(a1, a2) 被转化为 func(&a0, a1, a2)，以引用型参数形式传回返回值
  • 也就是说，如果使用了 0x0804f704 的函数片段， eax 的值会被修改成 3
  • 同样寻找 ebx 的片段
  • 发现 0x0806eb90 的非常契合我们的需要，因为能控制到 ebx, ecx, edx 的内容
  • 然后寻找 “/bin/sh” 字符串的地址
  • 还需要一个 int 0x80 的地址来触发系统中断
  • 参数够了，我们可以构造 payload 了

  • 这边假设我们不用 0x080bb196 来改 eax，用 0x0809ddda ，则需要多往栈上写 3 个无用参数

    • 也就是 pop eax ; pop ebx ; pop esi ; pop edi ; ret

ret2libc3

  • 文件 ret2libc3

  • 程序还是差不多，s 的偏移量还是一样的，只是这次没有 system，也没有 /bin/sh

  • 其实 system 函数属于 libc.so 这个动态链接库中的
    • 而这个库中函数之间相对偏移是固定的
    • 即使 Linux 开启了 ASLR 保护，地址最低的 12 位还是不会变化的
  • 如果知道了 libc 中某个函数的地址，可以借助这个地址在 libc 数据库中寻找对应版本的 libc.so
    • 然后我们就可以获取 system 函数的地址了

  • 如果需要知道 libc 中某个函数的地址，可以采用 got 表泄漏的方式

    • 由于 libc 的延迟绑定机制，我们需要泄漏已经执行过的函数的地址

  • 借助 LibcSearcher 可以快速的查找到对应版本 libc 的信息

  • 此外，在得到 libc 之后，其实 libc 中也是有 /bin/sh 字符串的，所以我们可以一起获得 /bin/sh 字符串的地址

  • 那么只需要泄漏 libc 中某个函数的地址，查出来对应 system 的偏移地址，还有 /bin/sh 的地址
    • 再次执行程序后栈溢出执行 system(“/bin/sh”) 就行了

  • 我们可以利用 puts 函数打印出 __libc_start_main 的地址，然后返回 _start 再执行程序，获取 shell

  • 构造 payload

中等 ROP

ret2csu

  • 在 64 位程序中，函数的前 6 个参数是通过寄存器传递的
  • 可以利用 x64 下的 __libc_csu_init 中的汇编片段，这个函数是用来对 libc 进行初始化操作
    • 一般可以在里面找到每个寄存器对应的汇编片段来调用
    • 不同版本的汇编片段可能不一样
  • 从 0x000000000040061A 一直到结尾，可以构造出 rbx, rbp, r12, r13, r14, r15 寄存器的数据

  • 从 0x0000000000400600 到 0x0000000000400609:

    • 可以将 r13 赋值给 rdx, 将 r14 赋值给 rsi，将 r15d 赋值给 edi

      • edi 其实就是 rdi 的低 32 位寄存器，但其实此时 rdi 的高 32 位寄存器值为 0

      • 而 64 位 gcc 编译的程序，函数传参时的前四个参数是 rdi, rsi, rdx, rcx

  • 如果可以合理地控制 r12 与 rbx，那么就可以调用我们想要调用的函数。

    • 在 0x0000000000400609 的地方有一个 call
    • 控制 rbx 为 0，r12 为存储我们想要调用的函数的地址

  • 从 0x000000000040060D 到 0x0000000000400614，可以控制 rbx 与 rbp 的之间的关系为 rbx+1 = rbp

    • 可以设置 rbx=0，rbp=1

改进

  • 要达成从 csu 溢出的目的，需要输入的字符串长达 128 位，但是不是所有情况都能输入这么多字符串的

  • 我们可以先控制 rbx 和 rbp，毕竟一个等于 0 一个 等于 1

  • 也可以在这两部分的 csu 片段中多次利用

  • 在上面的 libc_csu_init 中我们主要利用了以下寄存器

    • 利用尾部代码控制了 rbx，rbp，r12，r13，r14，r15。
    • 利用中间部分的代码控制了 rdx，rsi，edi。
  • 而其实 libc_csu_init 的尾部通过偏移是可以控制其他寄存器的。
    • 其中，0x000000000040061A 是正常的起始地址，
    • 可在 0x000000000040061f 处可以控制 rbp 寄存器
    • 在 0x0000000000400621 处可以控制 rsi 寄存器等等。
